AI卡片项目云侧方案。核心原则是：**主 Agent 做轻量 SOP 和候选选择，微服务做真实能力裁决、最终 CardSpec、DSL 生成与校验**。主 Agent 不承诺最终一定能生成某类动态卡片，等微服务返回后再组织用户回复。

# 端到端流程
```text
Step 1: App 中控根据 App 版本和 ROM 版本决定是否加载 Skill
        ROM < 7.0: 不加载本 Skill
        ROM >= 7.0: 允许 Skill Search / Skill 加载

Step 2: 主 Agent 判断用户 query 是否匹配卡片创建场景

Step 3: 主 Agent 调微服务获取能力概述
        返回数据能力概述、事件能力概述、素材概述

Step 4: 主 Agent 根据 query 筛选候选能力
        数据能力按需调用微服务加载完整 schema
        事件能力和素材能力可在 Step 3 直接全量返回

Step 5: 主 Agent 将 query、候选能力、候选 dataBindings、size 等信息传给微服务
        工具层自动携带用户 App 版本、ROM 版本、device、uid 等上下文

Step 6: 微服务生成卡片
        6.1 查询 IDS，获取用户实际安装应用、版本、意图/provider 列表
        6.2 过滤得到用户实际可用能力集合
        6.3 根据小艺 App 版本 + ROM 版本选择 A2UI 协议 profile
        6.4 生成最终 CardSpec，构造 A2UI 模型输入并生成 DSL
        6.5 校验产物；失败最多重新生成 1 次
        6.6 生成 artifact 文件，上传 OBS，返回链接和结构化状态

Step 7: 主 Agent 收到微服务结果，向用户回复
        success/degraded: 返回卡片生成结果标记
        unsupported/failed: 返回用户可理解说明和替代建议

Step 8: 端侧解析文件链接，下载 artifact，渲染 DSL 并持久化 DSL + CardSpec
```

其中App中控与主Agent能力框架本身已具备，云侧主要需要开发的是skill与微服务。


## 系统边界

### App 中控

职责：
- 根据 ROM/App 版本决定是否加载本 Skill。
- ROM < 7.0 时，不召回本 Skill。
- 将必要设备上下文放入会话或工具调用上下文。

不负责：
- 不做能力过滤。
- 不做DSL生成。

### 主 Agent / Skill

职责：
- 识别卡片创建场景。
- 调微服务获取能力概述。
- 根据用户 query 选择候选能力。
- 生成候选 dataBindings 或候选能力计划。
- 调微服务生成卡片。
- 根据微服务返回的状态组织自然语言回复。

不负责：
- 不维护完整 A2UI 协议。
- 不直接生成最终 DSL。
- 不承诺用户设备一定支持某项能力。
- 不生成最终 CardSpec。

### 微服务

职责：
- 维护能力注册表和版本兼容要求。
- 查询 IDS 获取用户实际设备能力。
- 过滤候选能力，得到实际可用能力。
- 生成最终 CardSpec。
- 根据版本选择 A2UI 协议 profile。
- 构造 A2UI 模型 system prompt 和 user messages。
- 调 A2UI 模型生成 DSL。
- 校验 DSL/CardSpec/artifact。
- 处理降级、拒答、失败重试。
- 上传 OBS，返回 artifact 链接和用户可读信息。

### A2UI 模型服务

职责：
- 根据微服务构造的输入生成 `genui` DSL。

不负责：
- 不查询用户设备能力。
- 不生成最终CardSpec。
- 不判断用户是否安装App。

### 端侧

职责：
- 解析 `genWidgetResult` 标记。
- 下载 artifact。
- 渲染 DSL。
- 用户确认后持久化 DSL + CardSpec。
- 卡片刷新时按 CardSpec 执行数据能力，向 DSL `updateDataModel` 写入数据。



# Skill 改造目标

当前 Skill 中很多细节可以下沉：

- 下沉到微服务：A2UI 协议、组件限制、美观规则、素材合法性、DSL 校验、重试修复、最终 CardSpec。
- 保留在 Skill：SOP 流程、能力概述筛选、候选 dataBindings 生成、工具调用、用户回复组织。
- Skill 不再直接输出 `genui/cardspec`，而是输出微服务返回的 `genWidgetResult` 链接和自然语言说明。

# 主 Agent 调微服务接口

作为工具插件形式供微服务调用，本工具内部分为3个能力，可以用不同入参来标识对应的能力。

1. `getWidgetCapabilityOverview`

输入：

```json
{
  "locale": "zh-CN",
  "appVersion": "x.x.x",
  "romVersion": "7.0.0"
}
```

输出：

```json
{
  "dataCapabilities": [
    {
      "id": "ViewWeather",
      "description": "查询当前天气、空气质量和未来预报"
    }
  ],
  "eventCapabilities": [ 
    (完整描述)
  ],
  "assetCandidates": [
    {
      "id": "asset.weather.rain",
      "description": "天气雨伞图标"
    }
  ]
}
```
说明：

- 数据能力只返回概述，不返回完整 schema。
- 事件能力和素材候选体积较小，可以在该接口中返回较完整信息。
- 待确认：最终方案返回json，或者返回一串让主Agent易于理解的普通文本形式。

2. `getDataCapabilitySchemas`

只针对数据能力做渐进加载。

```json
{
  "dataCapabilityIds": ["ViewWeather", "calendar.events.search"]
}
```

返回完整 `inputSchema/outputSchema/defaultWriteResultTo`。

最终方案建议不是返回json，而是返回一串让主Agent易于理解的普通文本形式。


3. `generateWidgetCard`

主生成接口。

```json
{
  "userQuery": "帮我做通勤卡片，包含天气和今日日程",
  "size": "2x4",
  "candidateDataBindings": [
    {
      "capabilityId": "ViewWeather",
      "arguments": {
        "districtName": "青浦区",
        "forecastDays": 1
      },
      "writeResultTo": "/data/weather"
    },
    {
      "capabilityId": "calendar.events.search",
      "arguments": {
        "timeRange": "today"

      "writeResultTo": "/data/calendar"

  "candidateEventCapabilities": [{"call":"clickToCallPhone","args":{"phoneNumber":{"path":"/contact/phoneNumber"}}}],
  "candidateAssetIds": ["asset.weather.rain", "asset.calendar.schedule"]

```
注意，candidateEventCapabilities不是传id，而是完整的语句(包括call与args，args中参数可以动态路径或直接静态写死)。

# 微服务核心模块

微服务从零构建，建议只保留必要模块：

```text
WidgetGenerationService
├─ CapabilityRegistry        能力注册表
├─ A2UIProtocolRegistry      A2UI 协议版本表
├─ DeviceCapabilityResolver  IDS 查询与能力过滤
├─ CardSpecBuilder           最终 CardSpec 生成
├─ PromptBuilder             A2UI 模型输入构造
├─ A2UIModelClient           DSL 模型调用
├─ Validator                 DSL/CardSpec/产物校验
├─ ArtifactStore             文件生成与 OBS 上传
└─ ResponsePlanner           降级/拒答/用户话术
```

不要一开始做复杂工作流引擎，普通服务编排即可。

# 能力过滤设计

微服务维护能力注册表：

```json
{
  "id": "calendar.events.search",
  "inputSchema": {},
  "outputSchema": {},
  "defaultWriteResultTo": "/data/calendar",
  "dependencies": {
    "minRomVersion": "7.0.0",
    "minAppVersion": "x.x.x",
    "requiredPackages": [
      {
        "packageName": "com.huawei.hmos.calendar",
        "minVersion": "14.0.0"
      }
    ]
  }
}
```

- defaultWriteResultTo非必选项。
- 数据能力、事件能力中，每个能力项都会对应dependencies，素材库整体无需dependencies，只与xiaoyi APP版本有关系

过滤顺序：

```text
能力 ID 是否注册
 -> ROM/App 版本是否满足
 -> 依赖 App 是否安装且版本满足
 -> IDS 返回的意图/provider 是否存在
 -> arguments 是否符合 inputSchema
```

注意：App 安装只是必要条件，不是充分条件。最终要以 IDS 返回的意图列表/provider 列表为准。可暂不考虑权限问题。

本期场景为一方app，所以只有运动健康可能会被卸载，对应能力不具备，其它能力理论上都会存在。

# CardSpec生成

主 Agent 传的是候选 `candidateDataBindings`。微服务过滤后得到 `effectiveDataBindings`，再生成最终 CardSpec。


```json
{
  "suggestSize": "2x4",
  "dataBindings": [
    {
      "capabilityId": "ViewWeather",
      "arguments": {
        "districtName": "青浦区",
        "forecastDays": 1
      },
      "writeResultTo": "/data/weather"
    }
  ]
}
```

如果没有有效数据能力，则静态 CardSpec 为：

```json
{
  "suggestSize": "2x2"
}
```

点击事件不进入CardSpec，只进入A2UI模型的候选事件输入。

# A2UI 协议版本兼容

微服务根据小艺 App 版本 + ROM 选择协议：

```text
 -> protocolProfileId
 -> 组件白名单、属性白名单、尺寸、绑定策略、事件策略、素材策略
```

协议对应若干个md文档。

# 校验与重试

校验对象是完整 artifact 文件，而不是只校验 DSL。

artifact 建议结构：

```json
{
  "schemaVersion": "widget-artifact-v1",
  "genui": "三行 JSONL",
  "cardSpec": {},
  "meta": {
    "protocolProfileId": "a2ui-form-rom7-v1",
    "capabilityRegistryVersion": "2026-07-03",
    "createdAt": 1780000000000
  }
}
```

校验项：

```text
genui 恰好三行 JSONL
createSurface/updateComponents/updateDataModel 顺序正确
root 尺寸与 size/profile 一致
组件、属性、事件、素材都在白名单内
children 引用可解析
updateDataModel.value 与模型输入一致
DSL 绑定路径可从 dataModel 或 CardSpec outputSchema 推导
CardSpec 只包含有效 dataBindings
CardSpec suggestSize 与 DSL 一致
```

失败后最多重新生成 1 次。重试 prompt 只追加错误摘要，不追加大量日志。

# 降级与拒答

这是体验关键。主 Agent 看到的是全量能力，微服务才知道真实能力，所以主 Agent 不要提前说“可以生成天气日历卡”。推荐话术是：

```text
我先检查一下当前设备支持情况，然后为你生成可用的卡片。
```

微服务内部状态需转为主Agent可感知的一系列描述。例如：

```json
{
  "status": "success | degraded | failed",
  "artifactUrl": "https://obs.xxx/widget/xxx.json",
  "userMessage": "日历权限未开启，已先生成只包含天气的通勤卡片。",
  "removedCapabilities": [
    {
      "id": "calendar.events.search",
      "reason": "PERMISSION_DENIED",
      "userReadableReason": "日历权限未开启"
    }
  ]
}
```

处理规则：

```text
核心能力可用
 -> 正常生成

部分能力不可用
 -> 降级生成，说明少了什么

组件样式不支持
 -> 不拒答，改写为支持组件表达

数据能力不存在
 -> 不伪造动态 CardSpec，可做静态卡/入口卡

App 未安装
 -> 有入口或静态价值则降级，否则提示安装/启用

所有核心能力不可用且静态卡无价值
 -> 不调用 A2UI 模型，直接 unsupported
```

拒答示例：

```text
当前设备上没有可用的外卖配送数据能力，也没有可打开的应用入口，所以暂时不能生成实时外卖卡片。你可以试试天气、日历或系统状态类卡片。
```

降级示例：

```text
日历权限当前未开启，我先为你生成只包含天气和通勤入口的卡片。开启日历权限后可以再生成包含今日日程的版本。
```

# 如何保证卡片好看

不要靠主 Agent。微服务从三处保证：

- 协议 prompt 内置美观规则：单一主焦点、少区域、受保护文本不截断、按钮贴底、色彩来源合法。
- 场景化素材：通过 `assetCandidates` 给模型可用素材白名单。
- 校验器增加视觉硬规则：区域数量、字号阶梯、间距阶梯、root shell、Row/Column 宽高预算、图片尺寸、禁用网络图。

生成失败或视觉风险高时，微服务强制降级为简单结构：

```text
标题 + 主状态 + 1 条支撑信息 + 1 个入口
```

# OBS 与端侧解析

微服务上传完整 artifact，不只上传 DSL。

返回：

```json
{
  "status": "success",
  "artifactUrl": "https://obs.xxx/widget/uuid.json",
  "artifactDigest": "sha256:xxx",
  "suggestSize": "2x4",
  "userMessage": "已为你生成通勤卡片。"
}
```

Skill 输出流式标记：

```text
```genWidgetResult:"https://obs.xxx/widget/uuid.json"```
```

端侧下载后拿到 `genui + cardSpec`，用于预览、确认添加和后续刷新。

# 开发任务拆

第一期最小闭环：

1. App 中控实现 ROM/App 版本分发，ROM < 7.0 不加载 Skill。 --
2. Skill 改造为工具编排型 SOP。
3. 微服务实现 3 个工具接口。
4. 能力注册表先用配置文件维护。
5. 接入 IDS 查询安装应用和意图列表。
6. 实现能力过滤、CardSpecBuilder、TaskSpec/PromptBuilder。
7. 接 A2UI 模型服务。
8. 实现规则校验器和 1 次重试。
9. 上传 OBS，返回 `genWidgetResult`。
10. 打通降级/拒答状态和用户话术。

这版不要一开始扩展过多数据能力或太细节的事项。先把一条动态卡片链路跑稳，再逐步补能力。先保证链路稳定、能力不伪造、卡片能渲染、失败能给用户合理回复。
